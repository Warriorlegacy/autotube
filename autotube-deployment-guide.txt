# AutoTube Deployment Guide

## Purpose
This guide explains how to deploy AutoTube in a clean, production-ready way using the architecture described in the product and technical requirements. AutoTube is designed as a modular Python pipeline orchestrated by `pipelinerunner.py`, with state stored in Supabase and the recommended scheduled trigger handled by GitHub Actions cron.[file:1][file:2]

## Deployment Model
For MVP, the preferred deployment model is GitHub Actions running on `ubuntu-latest`, because the TRD names GitHub Actions as the primary free scheduling option and includes a ready-made daily workflow that installs FFmpeg, restores the YouTube OAuth token, runs the pipeline, and uploads logs as artifacts.[file:1] The system is stateless at runtime, so the workflow can safely spin up, execute one pipeline run, write state back to Supabase, and exit.[file:1][file:2]

## Prerequisites
Before deploying, make sure the repository contains the modular Python codebase, config files, and directories described in the TRD, including `pipelinerunner.py`, module files under `modules/`, `config.yaml`, `.env.example`, `.gitignore`, and the output/log directories.[file:1] Also prepare the external services the pipeline depends on: Gemini or Groq for script and metadata generation, Edge-TTS for narration, Pexels and Pixabay for media, Supabase for state, Telegram for alerts, and YouTube Data API v3 for upload.[file:1][file:2]

## Required Secrets
Store all sensitive values as environment variables or GitHub Secrets rather than in code or committed files.[file:1] The TRD explicitly lists `GEMINIAPIKEY`, `GROQAPIKEY`, `PEXELSAPIKEY`, `PIXABAYAPIKEY`, `SUPABASEURL`, `SUPABASEKEY`, `TELEGRAMBOTTOKEN`, `TELEGRAMCHATID`, and the base64-encoded YouTube OAuth token as required secrets for CI/CD.[file:1]

## Local Setup
Use local setup first to validate the pipeline before enabling automation.[file:1] Create a virtual environment, install dependencies, install FFmpeg, copy the env template, complete the YouTube OAuth setup to generate `token.json`, and run unit tests plus a dry-run pipeline before scheduling uploads.[file:1] The TRD also recommends keeping generated media, logs, and secrets out of version control via `.gitignore`.[file:1]

## Supabase Setup
Create the Supabase tables for `topics`, `pipelineruns`, `youtubequota`, and optional `configoverrides` before the first production run.[file:1] The `topics` table stores the queue, the `pipelineruns` table records run history and failures, and the quota table tracks YouTube API unit usage so the pipeline can pause before hitting quota limits.[file:1][file:2]

## Config Setup
Populate `config.yaml` with the runtime defaults defined in the TRD, including target video duration, retry count, voice selection, media fallback behavior, video encoding settings, thumbnail template, quota thresholds, and alert preferences.[file:1] The configuration should remain overrideable via environment variables so you can change behavior without editing code.[file:1]

## GitHub Actions Workflow
Use a scheduled GitHub Actions workflow as the main deployment vehicle.[file:1] The TRD includes a daily cron workflow that checks out the repo, sets up Python, installs FFmpeg, installs Python dependencies, restores the OAuth token from a secret, runs `python pipelinerunner.py`, and uploads logs as artifacts.[file:1] Start with a private or dry-run mode until the pipeline is stable, then switch to the real upload schedule.[file:1][file:2]

## First Run Checklist
Before the first real scheduled run, verify that topic data exists in the queue, all secrets resolve correctly, Supabase connectivity works, and YouTube OAuth has been authorized successfully.[file:1][file:2] Confirm that the pipeline can generate a script, synthesize audio, fetch media, assemble a video, create a thumbnail, generate metadata, pass QA, and prepare an upload without manual intervention.[file:1][file:2]

## Operational Safety
Keep the first production uploads unlisted if you want an extra review layer, because the PRD supports an optional human review gate that uploads as unlisted and waits for approval before going public.[file:2] Respect the built-in QA checks, blacklist rules, AI disclosure requirements, and content-policy constraints so the system stays aligned with the project's compliance model.[file:1][file:2]

## Monitoring and Alerts
Enable Telegram alerts and structured logging from the start so failures are visible immediately.[file:1][file:2] The TRD expects stage-level logs in Supabase and Telegram notifications for success, failure, quota warnings, queue exhaustion, and weekly summaries.[file:1][file:2]

## Rollback Plan
If a deployment breaks, revert `config.yaml`, restore Supabase from backup if needed, or regenerate the YouTube OAuth token if it expires.[file:1] The TRD also recommends classifying errors as transient or permanent so the pipeline can retry safely when appropriate and alert immediately when not.[file:1]

## Scaling Notes
The MVP is intentionally built to stay inside free-tier limits, including Supabase storage, GitHub Actions minutes, and YouTube API quota.[file:2] If you later scale beyond the free-tier assumptions, the product docs outline clear upgrade triggers for hosting, voice quality, and storage.[file:2]

## Recommended Deployment Sequence
1. Clone the repo and create the Python virtual environment.[file:1]
2. Install FFmpeg and Python dependencies.[file:1]
3. Configure `.env`, `config.yaml`, and GitHub Secrets.[file:1]
4. Create the Supabase schema and seed the topic queue.[file:1][file:2]
5. Run tests and a dry-run pipeline locally.[file:1]
6. Restore the YouTube OAuth token into CI/CD.[file:1]
7. Enable the GitHub Actions cron workflow.[file:1]
8. Monitor logs, alerts, and quota usage for the first week.[file:1][file:2]
